③ Icon system — inconsistent inline SVGs everywhere

The codebase has ~30+ hand-drawn inline SVG icons scattered across blade files. They vary in size, weight, and visual style. There is no icon library loaded — no Heroicons, no Phosphor, no Tabler. This causes inconsistent visual weight across the UI.

Recommendation: Heroicons (already matches Laravel Breeze philosophy)

⑤ Missing features — what would meaningfully improve StudHub

Critical High Medium Low / v1.1
CRITICAL

Real PDF watermarking

The WatermarkResourceFile job currently generates an SVG stub, not a real per-user watermarked PDF. Install setasign/fpdi + tecnickcom/tcpdf. Without this, the anti-piracy promise to school admin cannot be kept.

HIGH

"Why was I notified?" explanation

When a routing notification fires, students don't know why. Adding one sentence — "You were notified because you're a BSIT-3 student who may have taken Data Structures" — dramatically increases response rates and trust in the routing engine. This is a 15-minute blade change.

HIGH

"I found it elsewhere" request resolution

Students often find resources via Messenger and forget to update StudHub. A "Resolved outside the system" button prevents the request board from filling with ghost open requests. One new action + status enum value.

HIGH

Empty states on all listing pages

The catalog, request board, shelf, and chat index show nothing when empty — no illustration, no CTA. First-time users will think the system is broken. Add a warm illustration + action prompt to each. The SEAIT-orange "Upload your first resource →" button would shine here.

HIGH

last_seen_at middleware

The routing engine ranks notification candidates by last_seen_at — but this field is never updated. All students appear equally active. Add a throttled middleware (max 1 write/minute per user) to update it on authenticated requests.

MEDIUM

Subject autocomplete in request form

Without autocomplete, students type wrong subject names and get unmatched requests. A Livewire wire:model.live + search query against subjects + subject_aliases tables solves this. ~30 minutes of work, enormous UX improvement.

MEDIUM

Student number field

The users table has no student_number column. For a school system, this is the canonical ID used by the registrar. Add it as nullable now, required after registrar sign-off. Allows history linking on re-enrollment.

MEDIUM

Cross-program resource flow widget (admin dashboard)

A widget showing "BSIT → BSCE: 12 resources this week" would be the most compelling demo slide for your panel. All data exists in the DB — it just needs a query + a simple bar or table component.

MEDIUM

Resource helpfulness signal ("thumbs up after download")

Karma is awarded for saves, but a save doesn't confirm the resource was useful. A simple post-download rating feeds into program_subjects.weight recalculation — this is the data flywheel that makes routing smarter over time.

MEDIUM

SoftDeletes on ChatMessage

When a moderator hides a message, $reported->delete() permanently removes it — even though the audit log still references it by ID. Add SoftDeletes to the model and add deleted_at column. One trait, one migration.

LOW / v1.1

PWA manifest + install prompt

The 00-product-overview.md says "Web + PWA only" — but there's no manifest.json or service worker. An installable PWA (icon on home screen) changes mobile engagement dramatically. ~1 hour of work.

LOW / v1.1

Dark mode

Tailwind's dark: variant + a <html class="dark"> toggle. Students studying at night will appreciate it. The current gray-100 background and white cards make for a very bright screen at midnight.

⑥ Additional UX improvements

📊

Student progress dashboard

Show a student's karma history as a sparkline, their uploaded vs. downloaded ratio, and a "you're in the top X% of contributors" badge. Motivates continued participation.

🔔

In-app notification bell

Current notifications go to email (digest) or Reverb push. A dropdown bell in the nav bar with unread count gives students instant feedback without leaving the app.

📱

Mobile-first responsive audit

The nav collapses on mobile (Alpine x-data "open" pattern is there) but the chat room and resource catalog aren't optimized for small screens. A 2-hour audit pass would cover the main flows.

🎓

Program-to-program request badges

Show the requesting student's program badge (e.g. "BSCE-2") on every request card. This immediately tells potential helpers whether the cross-program match is strong for them.

⚡

Onboarding checklist (first login)

After registration: a 3-step checklist — "Upload your first resource", "Ask for a resource", "Join a chat room." Drives first meaningful actions and reduces drop-off on day 1.

📖

"I can help with" subject tags

Let users declare which subjects they can assist with, separate from their uploaded resources. This feeds the routing engine more signal, and shows on public profiles as "Expert in: DSA, Networks".

Bottom line

StudHub is architecturally strong and genuinely well-built for a capstone. The three design gaps you identified — color scheme, logo, icons — are all a single day's work to fix and will transform the perceived polish of the system overnight. The color change alone (one config file + project-wide find-replace) will make it look like a different, far more professional product. The feature gaps are real but non-blocking for the pilot. Close the design debt first; the codebase quality is already there to be proud of.
